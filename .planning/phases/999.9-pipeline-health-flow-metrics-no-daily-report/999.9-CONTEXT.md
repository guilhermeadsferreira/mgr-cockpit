# Phase 999.9: Pipeline Health & Flow Metrics no Daily Report - Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

O daily report ganha uma seção de Pipeline Health que mostra saúde do fluxo por fase (Ready for Dev, Dev, To Do CR, CR, QA, Ready to Deploy) com tempo médio atual vs baseline histórico computado das done tasks. A função `categorizeStatus` é refinada para distinguir filas (To Do CR, To Do QA) de trabalho ativo (CR, QA). Seções de baixo valor ("O que fez ontem") são compactadas para reduzir ruído.

**Não inclui:** Dashboard UI separado (999.6), visão de PRs abertas (999.8), métricas persistidas em metricas.md além do que já existe (999.7).

</domain>

<decisions>
## Implementation Decisions

### Granularidade do Pipeline
- **D-01:** 6 fases explícitas no pipeline: Ready for Dev, Dev, To Do CR, CR, QA (agrupa To Do QA + QA), Ready to Deploy
- **D-02:** To Do QA e QA agrupados numa única linha porque a intervenção do gestor é a mesma (destravar QA). To Do CR e CR separados porque a intervenção é diferente (ninguém revisando vs. review complexo)
- **D-03:** `categorizeStatus` retorna granularidade fina: 'queue_dev', 'dev', 'queue_review', 'review', 'qa', 'queue_deploy'. O agrupamento para exibição acontece no `buildReport`, não no categorizador

### Baseline Histórico por Fase
- **D-04:** Baseline vem das done tasks do sprint atual — computar quanto tempo cada task done passou em cada fase via changelog (transições de status)
- **D-05:** Se não houver done tasks suficientes (< 3), mostrar "—" no baseline em vez de número enganoso
- **D-06:** Não requer nova persistência — dados são computados on-the-fly a partir do changelog das done tasks que já são carregadas via `getSprintIssues`

### Formato Pipeline Health
- **D-07:** Seção "## Pipeline Health" posicionada logo após a tabela de Sprint Overview e antes dos Alertas
- **D-08:** Tabela com colunas: `Fase | Tasks | Tempo médio | Mais antiga | Baseline (sprint) | Status`
- **D-09:** Status usa emoji: ✅ (tempo médio ≤ baseline ou ≤ threshold), ⚠️ (até 2x do baseline/threshold), 🔴 (> 2x)
- **D-10:** Thresholds default quando não há baseline: Dev 5d, CR 3d, QA 3d, Ready to Deploy 2d, Ready for Dev sem threshold (é fila natural)
- **D-11:** "Mais antiga" mostra o issue key + dias (ex: "CNT-858 (17d)") — link direto para o gargalo

### Compactação do "O que fez ontem"
- **D-12:** Substituir listing de commits individuais por contagem resumida: "3 commits em banking.transactional-schedule-api"
- **D-13:** Manter PRs merged e reviews como itens individuais (são acionáveis)
- **D-14:** Manter review comments substantivos (> 20 chars) — o contexto das discussões técnicas é valioso para o gestor

### Enriquecimento do Input para Haiku
- **D-15:** Passar dados de Pipeline Health (fase, tasks, tempo médio, baseline, status) como seção adicional no `analysisInput` para que a análise de IA possa gerar insights sobre gargalos do pipeline

### Claude's Discretion
- Ordem exata dos campos na tabela Pipeline Health
- Formatação dos números (arredondamento)
- Como lidar com fases que têm 0 tasks (omitir linha vs. mostrar "0")

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `enrichWithCycleTime()` — já busca changelog e computa daysInStatus por task. Pode ser estendido para computar tempo em cada fase (não só no status atual)
- `categorizeStatus()` — já mapeia status string para categoria. Precisa retornar granularidade fina (queue_dev, dev, queue_review, review, qa, queue_deploy)
- `getIssueChangelog()` — já retorna changelog completo com transições de status
- `computeDaysInCurrentStatus()` — helper que extrai última transição do changelog
- `DEV_STATUSES`, `REVIEW_STATUSES`, `QA_STATUSES`, `QUEUE_PATTERNS` — arrays de status já definidos

### Established Patterns
- O report é construído como array de strings (`lines[]`) no `buildReport()`
- Alertas são computados em `computeAlerts()` separado do build
- `batchParallel()` usado para limitar concorrência de API calls
- Análise de IA recebe `analysisInput` com 3 campos: sprintOverview, perPersonSummary, alerts

### Integration Points
- `buildReport()` — adicionar seção Pipeline Health entre Sprint Overview e Alertas
- `fetchSprintData()` — já carrega todas as issues do sprint. Done tasks estão disponíveis (statusCategory === 'done')
- `analysisInput` — adicionar campo `pipelineHealth` para o prompt Haiku
- `categorizeStatus()` — refinar retorno para granularidade fina

</code_context>

<specifics>
## Specific Ideas

- A tabela Pipeline Health deve ser a primeira coisa visualmente impactante após o sprint summary — o gestor abre o report e em 5 segundos sabe onde está o gargalo
- O usuário mencionou explicitamente: "task em Code Review mas a quanto tempo? Está em um tempo ok?" — o campo "Status" (✅/⚠️/🔴) responde isso diretamente
- Ready to Deploy precisa ser visível porque em fintech deploy tem janela restrita
- O alerta de WIP alto (B3) já foi corrigido nesta sessão para mostrar breakdown por fase — manter consistente com a nova categorização

</specifics>

<deferred>
## Deferred Ideas

- **Throughput trend** (tasks done por semana, tendência aceleração/desaceleração) — bom para dashboard 999.6, não para o daily
- **Queue time como métrica separada** no metricas.md — seria evolução do 999.7
- **CR review velocity** (quem está revisando, turnaround) — coberto parcialmente pelo 999.8 (PRs abertas)

</deferred>
