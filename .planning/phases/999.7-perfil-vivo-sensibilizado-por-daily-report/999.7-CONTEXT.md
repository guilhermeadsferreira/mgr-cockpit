# Phase 999.7: Perfil vivo sensibilizado pelo daily report - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Os report generators (Daily, Weekly, Sprint, Monthly) passam a persistir dados estruturados num arquivo `metricas.md` por pessoa — criando um retrato quantitativo rastreável que complementa o perfil qualitativo. O gestor abre o perfil e vê métricas evolutivas sem precisar ler relatórios avulsos.

Escopo: persistência de dados no metricas.md + seção "Momento Atual" + retenção automática. NÃO inclui: mudanças na UI, novos relatórios, ou alterações no perfil.md.

</domain>

<decisions>
## Implementation Decisions

### Dados a persistir por cadência
- **D-01:** Daily → só alertas ativos (blockers, WIP alto, cycle time warning). Dias normais não gravam nada.
- **D-02:** Weekly → métricas da semana (velocity em SP, PRs merged, reviews feitas, collaboration score, cycle time médio)
- **D-03:** Sprint → resultado consolidado (SP entregues vs planejados, cycle time médio, principais entregas, bloqueios)
- **D-04:** Monthly → tendência mensal (evolução vs mês anterior, destaques, pontos de atenção)
- **D-05:** Análise Haiku/IA fica SOMENTE no relatório — não persiste no metricas.md

### Estrutura do metricas.md
- **D-06:** Arquivo separado `metricas.md` por pessoa, ao lado do `perfil.md` em `pessoas/{slug}/metricas.md`
- **D-07:** Seção "Momento Atual" no topo — 3-4 linhas com tendências resumidas (velocity ↑↓↔, cycle time, code review, alertas ativos). Atualizada automaticamente pelo último relatório processado.
- **D-08:** Seções por cadência: `## Alertas Ativos`, `## Semanas`, `## Sprints`, `## Meses`
- **D-09:** Cada entrada mostra delta vs período anterior (ex: `velocity: 12 SP (+3 vs semana anterior)`)

### Retenção de histórico
- **D-10:** Últimas 12 semanas + últimos 6 sprints + últimos 6 meses. Entradas mais antigas descartadas automaticamente.
- **D-11:** Alertas ativos são limpos quando o alerta se resolve (não têm retenção fixa)

### Momento de execução
- **D-12:** Cada report generator (DailyReportGenerator, WeeklyReportGenerator, SprintReportGenerator, MonthlyReportGenerator) persiste no metricas.md como último passo após gerar o relatório. Sem job separado.

### Claude's Discretion
- Formato exato do YAML/Markdown dentro de cada seção do metricas.md
- Lógica de cálculo de delta e tendências (média móvel, comparação simples, etc.)
- Implementação da retenção (truncar no write ou cleanup periódico)
- Classe auxiliar para escrita no metricas.md (MetricsWriter ou inline nos generators)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ArtifactWriter` — Padrão de blocos gerenciados com comentários HTML (`<!-- BLOCO GERENCIADO -->`). Pode ser referência para como escrever/atualizar seções no metricas.md.
- `PersonRegistry.list()` — Retorna lista de pessoas com slug, relação, jiraEmail, githubUsername.
- `PersonDailyData` (interface no DailyReportGenerator) — Já tem todos os dados estruturados que o daily coleta por pessoa.

### Established Patterns
- Report generators recebem `workspacePath` no construtor e escrevem em `relatorios/`. A persistência no metricas.md seguirá o mesmo acesso a `workspacePath` mas escrevendo em `pessoas/{slug}/`.
- Blocos gerenciados no perfil.md usam comentários HTML como delimitadores. O metricas.md pode usar o mesmo padrão.
- `SettingsManager.load()` e `PersonRegistry` são instanciados dentro de cada generator — padrão já estabelecido.

### Integration Points
- Cada generator.generate() → adicionar chamada de persistência no final (após writeFileSync do relatório)
- `pessoas/{slug}/` — diretório já existe para cada pessoa cadastrada
- Scheduler já orquestra a execução dos reports — não precisa de mudança

</code_context>

<specifics>
## Specific Ideas

- O gestor quer abrir o perfil e ver um retrato atualizado sem ler relatórios avulsos
- Seção "Momento Atual" deve ser legível em 5 segundos — setas (↑↓↔) e números
- Deltas dão contexto imediato: "+3 SP vs semana anterior" é mais útil que "12 SP"
- O metricas.md deve servir como input para prompts de pauta e ciclo futuramente

</specifics>
