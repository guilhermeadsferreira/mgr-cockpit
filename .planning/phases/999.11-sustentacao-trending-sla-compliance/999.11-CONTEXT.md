# Phase 999.11: Sustentação — Trending histórico + SLA compliance rate - Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Acumular snapshots históricos do board de sustentação (não sobrescrever cache) e calcular métricas de tendência: SLA compliance rate (% tickets resolvidos no prazo), evolução de breach count e volume de tickets ao longo do tempo. Exibir deltas e mini gráficos de evolução na SustentacaoView existente.

Escopo: persistência de histórico, cálculo de compliance, visualização de tendência. NÃO inclui: integração com reports (999.12), intel por pessoa (999.13), alertas proativos (999.14).

</domain>

<decisions>
## Implementation Decisions

### Armazenamento de snapshots
- **D-01:** Snapshots acumulados em `history.json` no mesmo diretório de cache (`~/.pulsecockpit/cache/sustentacao/history.json`). Formato: array de snapshots com timestamp.
- **D-02:** Granularidade diária — um snapshot por dia. Se múltiplos fetches no mesmo dia, manter o mais recente (deduplica por data ISO sem hora).
- **D-03:** O `board.json` existente continua como cache de curto prazo (TTL 1h). O `history.json` é o acumulador de longo prazo, gravado a cada fetch bem-sucedido.

### Métricas de compliance
- **D-04:** SLA compliance rate = (tickets resolvidos dentro do threshold / total tickets resolvidos) × 100, por janela temporal.
- **D-05:** Duas janelas de compliance exibidas: 7 dias e 30 dias. A de 7d dá pulso rápido, a de 30d dá tendência.
- **D-06:** Para calcular compliance, usar tickets com `statusCategory === 'done'` e `resolved` dentro da janela. Comparar `ageDias(created)` vs threshold do tipo no momento da resolução (tempo total aberto, não tempo em status específico).

### Visualização de tendência
- **D-07:** Delta indicators nos cards existentes (Abertos, Fechados 30d, Breach): seta cima/baixo + valor absoluto vs snapshot de 7 dias atrás.
- **D-08:** Dois novos cards no topo: "SLA Compliance 7d" e "SLA Compliance 30d" com percentual e delta.
- **D-09:** Mini chart de evolução abaixo dos cards: breach count e compliance % ao longo do tempo (últimos 30 dias). Gráfico simples de linha.

### Retention de histórico
- **D-10:** 90 dias de snapshots diários retidos no `history.json`. Snapshots mais antigos são removidos automaticamente a cada gravação.
- **D-11:** Sem compressão de snapshots antigos — o volume é pequeno (1 snapshot/dia × 90 dias ≈ 90 objetos JSON).

### Claude's Discretion
- Biblioteca de gráfico para mini charts (recharts, visx, ou SVG inline)
- Formato exato do delta indicator (ex: "↑ 3" vs "+3 vs semana passada")
- Se compliance rate mostra 1 casa decimal ou inteiro
- Posição dos novos cards de compliance relativa aos existentes

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `fetchSupportBoardMetrics()`: já retorna SupportBoardSnapshot completo — reutilizar output diretamente como snapshot para history
- `fetchAndCacheSustentacao()` (index.ts): função privada que faz fetch + cache — ponto natural para adicionar gravação no history.json
- Cache dir já existe: `~/.pulsecockpit/cache/sustentacao/` — adicionar history.json ao lado de board.json
- MetricsWriter pattern de retention: `applyRetention()` que corta entradas além do limite — mesma lógica aplicável ao history.json

### Established Patterns
- Cache com writeFileSync/readFileSync (padrão do projeto — sem async fs)
- Graceful degradation: falha no history não impede retorno de dados (try/catch existente)
- IPC handlers retornam dados sem expor detalhes de persistência ao renderer
- SustentacaoView usa estado local com useEffect para fetch inicial

### Integration Points
- `fetchAndCacheSustentacao()` — após gravar board.json, adicionar lógica de append ao history.json
- `SustentacaoView.tsx` — novo IPC call para buscar history (ou incluir history no snapshot retornado)
- `ipc.ts` types — estender SupportBoardSnapshot ou criar novo tipo para dados com tendência
- preload/index.ts — expor novo método se IPC separado for necessário

</code_context>

## Specific Ideas

- O usuário quer saber "estamos melhorando ou piorando?" — deltas e tendência são o core value desta fase
- Compliance rate é a métrica que falta para transformar breach list (reativo) em indicador de saúde (proativo)
- History.json é pré-requisito para fases 999.12-999.14 — garantir que o formato suporta extensão futura

</specifics>
